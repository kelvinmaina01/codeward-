1.3
GitHub App registration & webhook receiver
BUILD
▶
Important: GitHub App vs OAuth App

You want a GitHub App, not an OAuth App. GitHub Apps install per-repo, have granular permissions, and can post check runs. OAuth Apps cannot post check runs. Do not confuse them.

Register the GitHub App
Go to github.com → Settings → Developer settings → GitHub Apps → New GitHub App
App name: Codeward (dev) — use (dev) suffix so you can register a production one later
Homepage URL: http://localhost:3000 for now
Webhook URL: use ngrok for local dev — ngrok http 3000 and paste the HTTPS URL + /webhooks/github
Webhook secret: generate a random string, save as GITHUB_WEBHOOK_SECRET in .env
Permissions → Repository: Contents (read), Checks (read+write), Pull requests (read+write), Metadata (read)
Subscribe to events: Push, Pull request, Check run
Generate a private key → download the .pem file → save contents as GITHUB_APP_PRIVATE_KEY in .env (replace newlines with \n)
Save App ID as GITHUB_APP_ID in .env
Build the webhook receiver
Install: pnpm add @octokit/app @octokit/webhooks
Create src/routes/webhook.ts — POST /webhooks/github route
Verify HMAC-SHA256 signature on every incoming webhook using X-Hub-Signature-256 header — REJECT any request that fails verification with 401
Parse the push event payload: extract repo full_name, commit SHA, branch, pusher, changed files list
Write a new run record to Supabase with status = 'pending'
Return 200 immediately — never make GitHub wait for processing
Post a check run back to GitHub
Use @octokit/app to authenticate as the GitHub App installation
After writing the run record, immediately post a check run with status='in_progress' and name='Codeward'
After 5 seconds (simulate), update the check run to status='completed', conclusion='success'
Push a test commit to your test repo and verify the yellow → green dot appears on the commit in GitHub

02
The Sandbox Core
Spin up a real isolated environment for each commit. Clone the repo, install deps, run a real database, run the test suite. This is the hardest phase. Get it right before building any agents on top of it.
⏱ Week 3–4 🎯 Customer test suite runs inside your sandbox Docker · Fly.io Machines · Node/Python detection
2.1
Docker base images — one per stack
BUILD
▶
Why base images matter

Every sandbox starts from a Docker image. If you install Node 20 + npm packages fresh every time, sandboxes take 3–5 minutes to boot. Pre-built base images with common deps cached reduce this to under 60 seconds. Start with Node only. Add Python later.

Node 20 base image
Create /docker/sandbox-node/Dockerfile — FROM node:20-slim, install git, curl, postgresql-client, install global tools: jest vitest
Add truffleHog v3 binary to the image: RUN curl -sSfL https://raw.githubusercontent.com/trufflesecurity/trufflehog/main/scripts/install.sh | sh -s -- -b /usr/local/bin
Add Trivy: RUN apt-get install -y wget && wget https://github.com/aquasecurity/trivy/releases/download/v0.50.0/trivy_0.50.0_Linux-64bit.deb && dpkg -i trivy*.deb
Add k6: download the k6 binary for linux/amd64 and add to /usr/local/bin
Build and push to Fly.io registry: fly auth docker && docker build -t registry.fly.io/codeward-sandbox-node:latest . && docker push registry.fly.io/codeward-sandbox-node:latest
Python 3.11 base image
Create /docker/sandbox-python/Dockerfile — FROM python:3.11-slim, install git, pip, pytest, same security tools
Build and push to Fly.io registry as codeward-sandbox-python:latest
2.2
Stack detector — auto-detect customer's tech stack
BUILD
▶
Create src/sandbox/stack-detector.ts — takes a list of repo files and returns a StackConfig object
Detection logic: if package.json exists → Node. If requirements.txt or Pipfile → Python. If Gemfile → Ruby. If go.mod → Go.

Return: {runtime: 'node', version: '20', testRunner: 'jest', database: 'postgres', dockerImage: 'codeward-sandbox-node:latest'}
Write unit tests for the detector with 5 different package.json fixtures
2.3
Fly.io Machines API — spin up and destroy sandboxes
BUILD
▶
Critical: always destroy machines

Every machine costs money per second. You MUST destroy it whether the run succeeds, fails, or times out. Wrap everything in a try/finally block. An uncaught exception that leaves machines running will bankrupt you.

Install Fly CLI: brew install flyctl (or curl install on Linux)
Create Fly app: fly apps create codeward-sandboxes
Get API token: fly auth token → save as FLY_API_TOKEN in .env
Create src/sandbox/fly-machine.ts with functions: createMachine(config), execInMachine(machineId, command), destroyMachine(machineId)
createMachine: POST to https://api.machines.dev/v1/apps/codeward-sandboxes/machines with image, env vars, resources (1 shared CPU, 1GB RAM for MVP)
destroyMachine: DELETE to /v1/apps/codeward-sandboxes/machines/{id}?force=true
Add a hard 10-minute global timeout using Promise.race([runPromise, timeoutPromise(600000)])
Wrap the entire run in try/finally — finally block ALWAYS calls destroyMachine regardless of outcome
Test: create a machine, exec echo "hello" in it, read the output, destroy it. Verify in Fly dashboard machine is gone.
2.4
Repo clone & test runner inside sandbox
BUILD
▶
Repo cloning
Use @octokit/app to generate an installation access token for the repo (expires in 1 hour — enough for a run)
Pass the token into the machine as an env var: GIT_TOKEN
First command in machine: git clone https://x-access-token:${GIT_TOKEN}@github.com/${repo}.git /workspace && cd /workspace && git checkout ${commitSha}
Dependency installation
After clone, run npm ci (not npm install — ci is faster and deterministic)
Capture stdout/stderr from npm ci — if it fails, mark run as failed with error "dependency install failed"
Test runner
Run the test command from StackConfig (e.g., npx jest --json --outputFile=/tmp/test-results.json)
Read /tmp/test-results.json from the machine after completion
Parse results: extract numPassed, numFailed, numTotal, and array of failed test names
Store parsed results in Supabase run_results table
Update GitHub check run: if numFailed > 0, set conclusion='failure' with summary listing failed tests
Test with a real repo that has passing tests → verify all pass → introduce a deliberate failing test → verify GitHub shows failure with correct test name

⏱ Week 5–6 🎯 Claude detects a real secret in a test repo and blocks the merge Anthropic SDK · Claude Haiku 4.5 · tool_use · truffleHog · Trivy
3.1
Anthropic SDK setup & tool_use pattern
BUILD
▶
Do NOT use LangChain or any agent framework

Use the Anthropic SDK directly. It is simpler, faster, cheaper, and you understand every call. One file, one pattern, full control. LangChain adds abstraction that hides what the model is actually doing.

Install: pnpm add @anthropic-ai/sdk
Add ANTHROPIC_API_KEY to .env — get from console.anthropic.com
Create src/agents/base-agent.ts — a reusable function that runs a Claude agent loop with tool_use
The agent loop: send message to Claude → if response has tool_use blocks → execute each tool → add tool result to messages → send again → repeat until no more tool_use
Always pass max_tokens: 4096 and model: "claude-haiku-4-5" for security/bloat agents
Add token counting — log input tokens, output tokens, and estimated cost per run
Write a simple test: ask Claude to use a mock tool that returns "Paris" for "capital of France" — verify the loop works
3.2
Security Agent — 3 checks first
BUILD
▶
Tool 1: run_trufflehog
Create tool definition: run_trufflehog — input: {workspacePath: string}. Runs truffleHog inside the Fly machine on /workspace. Returns JSON array of findings.
Tool executor: exec trufflehog filesystem /workspace --json --no-update 2>/dev/null in the machine. Parse each line as JSON (truffleHog outputs NDJSON).
Return structured array: [{detector: "AWS", file: "config.js", line: 14, raw_secret_partial: "AKIA...xxxx"}]
Tool 2: run_trivy
Create tool: run_trivy — exec trivy fs /workspace --format json --quiet. Parse output. Return array of CVEs with severity, package name, installed version, fix version.
Tool 3: probe_auth_routes
Create tool: probe_auth_routes — starts the app in the machine, discovers routes from express/hono route list, fires GET/POST to each with no Authorization header. Returns list of routes that returned 200 (should have returned 401).
Wire into Security Agent
Create src/agents/security-agent.ts with system prompt: "You are a security agent. You must use all three tools and report every security issue you find. Return a JSON report."
First user message: "Analyse the codebase at /workspace for security issues. Repo: {repoName}. Commit: {sha}."
After agent completes, parse the final JSON report and store in run_results with category='security'
If any finding has severity='critical', set run status='blocked' and update GitHub check run to conclusion='failure'
Test it
Create a test repo with a hardcoded fake AWS key: const key = "AKIAIOSFODNN7EXAMPLE"
Push a commit. Verify: GitHub check run turns red, run is blocked, Supabase shows security finding with file and line number.
🔒 Phase 3 Gate — Pass ALL before Phase 4
Secret detected: A repo with a hardcoded fake AWS key gets blocked. The check run shows the file name and line number of the key.
CVE detected: A repo with lodash@4.17.15 in package.json gets a warning with the CVE ID listed.
Clean repo passes: A repo with no secrets and no critical CVEs gets a green check run. Not everything is flagged.
Cost check: Log the Anthropic API token usage per run. One run should cost under $0.05 in API fees using Haiku 4.5.
Tool loop test: Add a console.log inside each tool executor. Verify all 3 tools are actually called by Claude on every run (not skipped).

AST Engine + Bloat Agent
Parse code into an Abstract Syntax Tree. Find duplicate functions. Generate annotated diffs with reasons. The bloat agent is what makes Codeward unique — no other tool does this automatically.
⏱ Week 7–8 🎯 Duplicate function found, refactored, diff posted to PR with reason tree-sitter · Claude Haiku 4.5 · GitHub PR comment API
4.1
tree-sitter integration in sandbox
BUILD
▶
Install in sandbox image: npm install -g tree-sitter-cli and add to Docker base image
In the agent-runner app, install: pnpm add tree-sitter tree-sitter-javascript tree-sitter-typescript
Create src/ast/parser.ts — takes a file path, returns a parsed tree with all function declarations and their bodies
Create src/ast/duplicate-finder.ts — takes list of parsed trees, returns pairs of functions that have the same body hash (SHA256 of normalised body text)
Test with 2 files that have the same validateEmail function — detector should return them as a duplicate pair
4.2
Bloat Agent — 3 checks first
BUILD
▶
Create tool: find_duplicates — runs duplicate-finder.ts on all JS/TS files in /workspace. Returns array of duplicate pairs with file, function name, line numbers.
Create tool: find_dead_code — trace import/export graph in the repo. Return functions/vars that are exported but never imported anywhere.
Create tool: generate_refactor — given a duplicate pair, Claude generates the refactored code: removes the duplicate and adds an import from the canonical location.
Bloat Agent system prompt: "You are a refactoring agent. Find all duplicate code and dead code. For each duplicate, generate a precise refactor. Explain your reasoning for every change."
After agent runs, parse its output into a structured diff: [{file, original_lines, refactored_lines, reason}]
Store the diff JSON in Cloudflare R2 keyed by run ID
Post the diff as a PR comment on GitHub using the checks API — show file, old code (red), new code (green), reason (blue)
In suggest mode: diff is shown but not applied. In auto mode: apply the diff to a new commit on the branch.
🔒 Phase 4 Gate
Duplicate found: A repo with the same function in 2 files — agent finds it, generates a refactor, posts the diff to the PR comment with the reason.
Tests still pass: After the refactor is applied (auto mode), run the test suite again. All tests still pass.
No false positives: A repo with no duplicates — agent says "no bloat found." Does not invent issues.

08
Codeward AI Chat Agent
The always-on agent in the sidebar. It reads real run history, can spawn agents on demand, answers questions about your codebase, and takes action with your approval. This is what makes Codeward feel like a senior engineer, not a tool.
⏱ Month 4–5 🎯 User asks "what is our biggest risk?" and agent answers with real data Claude Sonnet 4.5 · WebSocket streaming · pgvector semantic search
This is your moat — build it carefully

The chat agent is what separates Codeward from every other CI/CD tool. It is not a chatbot. It has tool access to read actual run data, spawn real agents, and take real actions. The key is: it never guesses. When asked "what is broken in payments-api?" it calls the query_run_history tool, reads the last 5 run reports, and answers from real data. Build this trust mechanism carefully — it is what gets teams to Enterprise tier.

🔒 Phase 8 Gate — and YC readiness check
Data-grounded answers: Ask "what are the 3 biggest risks in my codebase?" The agent reads real run data and answers with specific files, lines, and findings — not generic advice.
Action test: Tell the agent "fix the N+1 in payments-api." It spawns the architecture agent, runs it, returns the diff, and waits for your approval before applying.
5 real teams: By this point you have 5 real engineering teams using Codeward. At least 1 has been using it for 30+ days. You have evidence of debt reduced over time.
1 paying customer: At least 1 team has upgraded from free to Pro ($49/month). They pay because the auto-refactor saved them real time.
YC application ready: You can answer "what has actually been built?" with: running product, real users, real data, measurable debt reduction numbers.
