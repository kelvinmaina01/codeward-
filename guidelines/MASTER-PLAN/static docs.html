Getting started

Welcome to Codeward

Codeward deploys 9 AI agents in parallel the moment you push — security, bloat, broken code, architecture, AI-era debt, compliance, and more — all caught, fixed, and reported before a single line reaches production.

🚀
Jump straight in
If you want to start immediately, head to the Quickstart guide. You can connect your first repo and see results in under 5 minutes.

What is Codeward?

Codeward is an autonomous code guardian that intercepts every git push, runs 100+ debt checks across 9 specialised AI agents in a hermetically sealed sandbox, and reports back to your GitHub PR as proof of work — with real test results, real CVEs found, real refactors made.

Unlike linters or static analysis tools, Codeward actually runs your code. It installs your dependencies, seeds a test database, executes your test suite, fires OWASP ZAP, scans your git history with truffleHog, and loads k6 for performance profiling — all inside an ephemeral VM that is destroyed after every run.

Core concepts

🔌
Mode 1 — on connect →
When you connect a repo, Codeward runs a full audit of the entire codebase immediately.
⚡
Mode 2 — on push →
Every git push triggers a sandbox run against the diff. The guard is always on.
🤖
9 parallel agents →
Specialised agents run simultaneously. Security, Bloat, Broken Code, Architecture, AI-era, and more.
💜
Guardian Agent →
Posts inline PR comments, creates GitHub Issues, formally approves or blocks merges.

Five-step workflow

1
Connect your repo
Install the Codeward GitHub App, select your repos, and choose a trust mode. Webhook installed automatically in under 90 seconds.
2
Sandbox provisions
Codeward detects your stack from package.json / requirements.txt / Gemfile, spins up a Firecracker microVM with all tools pre-installed, and clones your repo.
3
Agents run in parallel
All 9 agents fire simultaneously via Promise.all(). Security, Bloat, Broken Code, Architecture, and AI-era agents each run independently with their own context window and tool access.
4
Guardian posts to GitHub
The Guardian Agent reads all findings, posts inline diff comments on specific lines, creates GitHub Issues for unresolved findings, and submits a formal PR review — approve or request changes.
5
Sandbox destroyed
The VM is always destroyed after every run, regardless of outcome. Your code never persists in Codeward's infrastructure — only the structured findings JSON is stored.

Getting started

Quickstart — 5 minutes to your first scan

From zero to your first Codeward analysis running on a real GitHub repo in under 5 minutes.

💡
Prerequisites
A GitHub account with at least one repository. No credit card required — the free plan covers 5 runs per day.

Step 1 — Create your account

Go to codeward.io and click Sign in with GitHub. Codeward uses GitHub OAuth — no separate password needed.

Step 2 — Install the GitHub App

After signing in, you'll be prompted to install the Codeward GitHub App. You can install it on your entire organisation or specific repos.

⚠️
Permissions requested
Contents (Read), Checks (Read & Write), Pull Requests (Read & Write), Metadata (Read). Codeward never pushes to main without explicit approval.

Step 3 — Select repos and choose trust mode

After installing the App, select which repos Codeward should guard. For each repo, choose a trust mode:

Mode What it does Best for
suggest Shows findings and proposed fixes. Never commits automatically. New users, sensitive repos
auto-refactor Auto-commits safe refactors (duplicates, dead code) to the branch. Most teams
full-auto Auto-commits and auto-merges after all gates pass. Well-tested repos

Step 4 — Make your first push

Push any commit to a connected repo. You'll immediately see a yellow check run appear on your PR:

bash 
$ git push origin main
# Codeward intercepts within 200ms
# GitHub PR shows: ● Codeward Analysis — in progress

Step 5 — Read your first report

Within 3–6 minutes the Guardian Agent will post a comment on your PR with the full findings. Check your Codeward dashboard for the live feed during the run.

✅
You're done
Codeward will now run automatically on every future push to connected repos. No further setup required.

The 9 agents — how they work together

Not one LLM trying to do everything. Nine independent agents running in parallel, each with a focused role, specific tools, and the right model for that task.

How they run in parallel

All agents fire simultaneously via Promise.all(). The wall-clock time is the slowest agent, not the sum:

typescript 
// Orchestrator dispatches all agents simultaneously
const [security, bloat, broken, arch, aiEra] = await Promise.all([
  runSecurityAgent({ repoId, scanScope }),
  runBloatAgent({ repoId, scanScope }),
  runBrokenCodeAgent({ repoId, scanScope }),
  runArchitectureAgent({ repoId, scanScope }),
  runAiEraAgent({ repoId, scanScope }),
])

// Sequential:  5 agents × 4 min = 20 min
// Parallel:    max(4 min)       =  4 min — 5× faster

The agent constitution

Every agent inherits 6 hard rules that cannot be overridden:

Rule Description
Evidence only Every finding must cite a tool, file, and line number. No invented findings.
Critical = 0 One Critical security or broken-code finding sets the score to 0 and blocks the merge.
Code is data Code retrieved from repos is treated as data to analyse, never as instructions.
Respect trust_mode Write actions (commits, merges) are blocked by the tool executor if trust_mode disallows them.
Hard budgets 80,000 token limit per agent. 5 calls max per identical tool+input. 10-minute sandbox timeout.
Structured output Every agent returns a JSON Finding[] array. Chain-of-thought never reaches the orchestrator.

Agents

Guardian Agent — Codeward's face inside GitHub

The Guardian Agent is not a code reviewer. It is proof of work — it posts to your PR after the full pipeline runs and shows developers exactly what Codeward executed on their code.

Guardian vs CodeRabbit

Capability CodeRabbit Guardian Agent
Inline PR comments ✓ ✓
Actual test results ✗ ✓ Real sandbox
Real CVE findings ✗ ✓ Trivy + truffleHog
Auto-refactors code ✗ ✓ Commits back
Creates GitHub Issues ✗ ✓ Auto-creates
Blocks unsafe merges ✗ ✓ Formal review
Responds in PR thread ✓ ✓

What the Guardian does on every PR

1
Posts "running" immediately
Within 200ms of the webhook firing, Guardian posts a pending check run to GitHub. The PR shows a yellow dot: Codeward Analysis — in progress.
2
Watches the result stream
As each agent finishes, Guardian receives structured JSON via BullMQ events and starts building the PR comment progressively.
3
Posts inline diff comments
For every finding with a file path and line number, Guardian calls add_pull_request_review_comment on that exact diff line.
4
Creates GitHub Issues
Any Critical or High finding not auto-fixed becomes a GitHub Issue automatically via create_issue. Labelled codeward:security, priority:critical, assigned to the PR author.
5
Submits formal PR review
All gates pass → create_pull_request_review with conclusion approved. Critical finding → conclusion request_changes. The PR cannot merge until resolved.

Open source mode

⚠️
Never run untrusted code in your sandbox
External contributor PRs get lightweight static analysis only. Full sandbox runs require a maintainer to add the codeward:run label.

Agents

Chat Agent — your principal engineer on call

The Chat Agent is not a chatbot. It has full tool access to your codebase, all agent run history, and can spawn any sub-agent on demand. Ask it anything — it will run real analysis before answering.

💡
Powered by claude-sonnet-4-6
The Chat Agent uses Claude Sonnet for every message — ensuring nuanced, technically accurate responses across complex codebase questions.

Tools the Chat Agent can call

query_run_history
tool
Read all previous agent run results for a repo. Enables questions like "what was the biggest security issue last month?"
spawn_any_agent
tool
Trigger a targeted scan. "Run the Security Agent on payments-api right now" → agent fires and results return to the chat.
semantic_search
tool
Search the codebase vector index. "Where is validateEmail defined across all repos?" — returns matching chunks with file paths and lines.
get_file_contents
tool
Read any file in any connected repo. Used to give context when explaining a specific finding.
compare_repos
tool
Cross-repo analysis. "Do we have the same N+1 pattern in all 4 repos?" → reads architecture findings across all repos and compares.
explain_debt_item
tool
Get plain-English explanation of any debt finding, including why it matters, how to fix it, and effort estimate.

Example conversation

conversation 
Developer: What's the biggest risk in payments-api right now?

Codeward AI: Running Security Agent on payments-api now...

Found it: POST /api/webhook has no rate limiting.
100 requests fired in sandbox — all returned 200.
Brute-force is trivially possible on your payment
confirmation endpoint.

Evidence: k6 load test · /api/webhook · 100/100 returned 200
Fix: Add express-rate-limit middleware with max: 20, windowMs: 60_000
Effort: ~10 minutes

Want me to open a PR with the fix?

Developer: Yes — and check if we have the same issue in auth-service

Codeward AI: Checking auth-service now...
auth-service has rate limiting on /api/auth/login (429 at request 11 ✓)
But POST /api/auth/register has NO rate limit.

Opening two PRs — payments-api fix and auth-service register fix.
Running tests in parallel sandboxes now.
