Yes — this is a multi-agent system. 9 specialised agents, not one AI doing everything.

Why multi-agent and not one big prompt

A single LLM trying to do security scanning, AST refactoring, compliance checking, and PR review in one prompt would be slow, expensive, and bad at all of them. Specialised agents with tool access are faster, cheaper, and measurably better. Each agent has a focused role, specific tools, and the right model for that task. This is the same pattern Anthropic uses internally.

How agents communicate — the message bus

Agents never call each other directly. They communicate through a shared job queue (BullMQ on Redis) and a structured result store (Postgres).

1 — Orchestrator publishes
Writes job to Redis queue with commitSHA, repoId, priority, agentList. Each sub-agent picks its own job independently.
2 — Sub-agents run in parallel
Security, Bloat, Broken Code, Architecture, AI-Era all run simultaneously inside the same sandbox. Wall-clock time: 4–6 min total.
3 — Guardian reads results
Guardian Agent subscribes to the result stream. As each agent finishes, Guardian receives structured JSON and starts building the PR report — no waiting for all agents to finish.

Orchestrator
claude-sonnet-4-6

claude-haiku-4-5 + deterministic tools
Most checks are deterministic tool calls (ZAP, truffleHog, CVE lookups). LLM interprets output and generates fix suggestions. Haiku is fast and cheap for this.

EU AI Act reasoning, prompt injection testing, and RAG freshness require nuance. Sonnet handles the ambiguity. Haiku would miss edge cases.
Guardian Agent

Writes developer-facing prose that will be read by real engineers. Must be clear, precise, and technically credible. Cannot afford Haiku's occasional imprecision here.
Chat Agent

Conversational, technically deep, spawns other agents mid-conversation. Sonnet is best for this pattern. Opus would be too expensive per message.

Use the Anthropic SDK directly with native tool_use. Every abstraction hides bugs and costs you money you cannot debug. You want to see every API call your system makes.

Codeward Guardian — what it does
Everything CodeRabbit does — PLUS all of this:
Posts PR summary showing REAL sandbox test results
Shows actual CVEs found with severity and fix version

Posts debt score 0–100 with per-category breakdown
Creates GitHub Issues for every unresolved finding
Formally blocks the PR if critical security found

Is the proof of work of what Codeward actually did
The key positioning difference

CodeRabbit is a code reviewer. The Guardian Agent is a proof of work. It posts AFTER the full pipeline — sandbox, all agents, all 100+ debt checks — and shows developers exactly what Codeward actually executed on their code. That is a fundamentally different value proposition.

Step-by-step flow — what the Guardian does on every PR

01
PR opened → immediately posts "running" status
Guardian fires instantly via webhook. Posts to PR: "Codeward analysis running…" with a pending check run (yellow dot on the PR). Developer sees it immediately — no silent waiting.
02
Watches the pipeline result stream
As each sub-agent (Security, Bloat, Broken Code etc.) finishes, Guardian receives structured JSON via BullMQ events. It does not wait for all agents — it starts building the report progressively.
03
Posts inline comments on specific diff lines
For every finding with a file path and line number, Guardian posts an inline review comment directly on that line of the diff — exactly like CodeRabbit. "Line 42: hardcoded API key detected and moved to env var. sk-live-xxx → process.env.STRIPE_KEY"
04
Creates GitHub Issues for unresolved findings
Any Critical or High finding that was NOT auto-fixed becomes a GitHub Issue automatically via the MCP create_issue tool. Title, description, suggested fix, severity label, assigned to the PR author. Team can track on the project board.
05
Submits formal PR review — Approve or Request Changes
If all gates pass: create_pull_request_review with conclusion "approved" and summary showing the full score. If blocked: conclusion "request_changes" with the exact critical finding. The PR cannot merge until resolved. This has real weight — it shows up as a required review.
06
Responds to comments in the PR thread
Developer replies: "Why did you remove this function?" Guardian reads the thread context and replies with the full reasoning — references the duplicate function, shows the line it was merged into. A full conversation, not a one-way report.
07
Updates after refactors are applied
If Bloat Agent commits a refactor back to the branch, Guardian updates its PR comment: "Refactor applied — validateEmail() merged into utils/validators.js. Tests re-run: 142/142 passing."

Open source mode — two-tier trust model

Never run untrusted code in your sandbox

Open source repos get PRs from strangers. An attacker can craft a PR that exfiltrates your env vars during sandbox execution. GitHub built Lockdown mode in their MCP server for exactly this reason. You need the same protection.

External contributor PRs — lightweight only

Static AST analysis of the diff only (no execution). Secret detection via truffleHog (read-only). CVE audit of lockfile changes (no execution). Duplicate detection against existing codebase. Guardian posts: "Quick scan complete. Full sandbox requires maintainer label: codeward:run"

After maintainer adds codeward:run label

Full sandbox fires. All agents run. Guardian posts complete report. Maintainer reviews findings and merges or requests changes. Any accepted contribution's medium issues become GitHub Issues automatically for tracking.

Open source = your fastest viral growth channel

Every popular open source project that installs Guardian gets a "Codeward Verified" badge in their README. Every contributor sees Guardian's comments. Every maintainer who loves it installs it at their day job. Product-led growth without ad spend.

GitHub MCP — every action available and how each agent uses it

The GitHub MCP server gives agents access to GitHub's full API through structured tool calls. Authentication via your existing GitHub App (production) or fine-grained PAT (development).

Read actions — safe, no approval needed

get_file_contents
Bloat + Security agents read any file in the repo. Full codebase scan without cloning locally. Used for cross-repo duplicate detection.
list_commits / get_commit
Audit trail. truffleHog-style scan across git log for secrets committed months ago. Compliance agent checks commit authors for NHI compliance.
get_pull_request
Guardian reads full PR diff, title, description, reviewers before building its comment. Understands context of what changed and why.
list_dependabot_alerts
Security Agent reads GitHub's own CVE alerts and cross-references with Trivy scan. Double confirmation before reporting a CVE.
get_workflow_run_logs
DX Agent reads CI/CD failure logs. "Why did the release.yml job fail?" — real answers from real logs, not guesses.
search_code
Chat Agent answers "where is validateEmail defined across all repos?" Cross-repo semantic search without cloning everything.
get_code_scanning_alerts
Architecture Agent reads existing SAST findings so Guardian doesn't duplicate them in the report.
list_issues
Guardian checks for existing issues before creating duplicates. Chat Agent answers "what are our open security issues right now?"
list_pull_requests
Compliance Agent monitors open PRs for unreviewed security-sensitive changes. Dashboard shows PR health across all repos.
get_repo_content (tree)
Stack detector reads directory structure before spinning up sandbox. Detects framework, package manager, test runner without cloning.

Write actions — require trust mode setting

create_pull_request_review Core Guardian action
Formal PR review — Approved or Request Changes — with inline diff comments. Has the same weight as a human reviewer. PR can be configured to require this review before merging.
add_pull_request_review_comment Core Guardian action
Inline comment on specific diff lines. "Line 42 — hardcoded key detected." Developer replies. Guardian replies back. Full conversation thread inside GitHub.
create_issue Auto on Critical/High
Every unresolved Critical or High finding becomes a GitHub Issue automatically. Title, body, labels (codeward:security, priority:critical), assigned to PR author.
add_labels_to_issue Auto-labelling
Labels created issues: codeward:security, codeward:bloat, codeward:architecture, priority:critical/high/medium. Board stays organised automatically.
create_or_update_file Auto-refactor mode only
Bloat Agent commits refactored files back to the branch in auto-refactor mode. Commit message: "refactor: Codeward — removed 3 duplicates, fixed 1 security issue"
merge_pull_request Requires explicit permission
After staging approval. Only fires after ALL gates pass AND human approves. Never auto-merges without explicit .codeward.yml permission setting.
create_branch Audit mode
For existing repo full audits — creates a codeward/audit-fixes branch with all proposed refactors as a single PR for the team to review all at once.
GitHub Projects API Enterprise only
Compliance Agent adds debt items to the team's project board. Creates "Codeward Debt" column. Tracks resolution across sprints. Requires projects toolset enabled.

Auth — PAT vs GitHub App

Development (MVP)
Fine-grained Personal Access Token (PAT)
User generates a PAT at github.com/settings/personal-access-tokens with exactly the permissions needed. Set as GITHUB_PAT env var. The GitHub MCP server accepts it directly. Limitation: scoped to one user's repos.
Production
GitHub App (what you already built)
Authenticates as the app. Gets per-installation tokens. Acts across all connected repos without needing each user's PAT. This is the correct production pattern. Your existing GitHub App registration already gives you this.

Two user types — completely different product experiences

This is a critical product architecture decision you must get right at onboarding

These two users need different entry points, different first experiences, and different value propositions. Building only for one will confuse the other and kill retention.

User Type A — Active developers (prevention mode)
They are coding right now. Repos have recent commits. They need a guardrail on every future push.
Who they are

Solo founders on Lovable/Bolt. Dev teams using Cursor daily. Startups shipping fast. Repos with commits in the last 7 days. They are creating debt right now and need it stopped.

What they need

Prevention. Every push is a gate. Sandbox runs on every commit. Guardian reviews every PR. Health score rises week over week. They want the guardrail, not an audit.

1
Sign up → OAuth → select repos → webhook auto-installed → 90 seconds total
2
Next push → sandbox runs → Guardian posts inline PR comments → debt score visible
3
Iterate: each push is cleaner — health score rises, debt items removed shown in dashboard
4
Dashboard shows: "your codebase is 14% leaner than 30 days ago" — proof of value
Push-drivenPrevention modeWebhook always on
User Type B — Existing codebases (audit mode → then guard)
They have months or years of existing code. They need a full audit NOW before going into guard mode.
Who they are

CTOs inheriting AI-generated codebases. Teams that shipped fast for 6 months. Companies preparing for due diligence or a security audit. Repos with 50+ commits, 200+ files, last commit could be any time.

What they need

A full retrospective scan of the entire repo as it stands today — not just the last commit. A prioritised fix plan. GitHub Issues for every finding. Then ongoing guard mode.

1
Sign up → OAuth → connect repo → onboarding detects "existing codebase" signals → shows "Full Audit" option
2
User clicks "Audit everything first" → Codeward clones full repo (HEAD commit) into sandbox

4
Full audit report: health score, all findings, prioritised fix plan with effort estimates
5
Guardian creates GitHub Issues for every finding — team triages on the board, assigns to sprints
6
Guardian creates a single codeward/audit-fixes branch PR with all auto-fixable issues as one reviewable change
7
After audit complete → guard mode activates automatically — every future push is protected

How to detect which mode to offer at onboarding

Auto-detection signals

Last commit age: More than 7 days ago → suggest Full Audit first
Commit count: More than 50 commits → not a new project → offer full scan
File count: More than 200 files or 20,000 lines → substantial codebase → Full Audit
No webhook history: Repo existed before Codeward install → always offer audit first

Always show both options explicitly

Onboarding choice screen:
Guard from now →
Start protecting every future push immediately.
Audit everything first →
Scan the full repo now, then guard going forward.

Full tech stack — every choice with the honest reason why

Frontend — React

Framework
React 18 + TypeScript + Vite
React 18 concurrent rendering handles the live run feed without blocking the UI. Vite is 10× faster than CRA for dev builds. TypeScript is non-negotiable — a product about code quality must eat its own cooking.
Styling
Tailwind CSS v4 + shadcn/ui
shadcn/ui gives production-quality components (dialogs, tables, dropdowns) without a design team. You own the components — no version lock. Tailwind v4 is significantly faster to build with.
State
Zustand + TanStack Query
Zustand for UI state (current screen, selected repo). TanStack Query for server state — handles caching, background refetch, and optimistic updates. Perfect for the dashboard polling pattern.
Real-time
Socket.io client
Live run feed needs WebSocket. Socket.io is the simplest self-hosted option. Streams BullMQ job events to the dashboard in real time.
Code display
Monaco Editor (read-only) + react-diff-view
Monaco is the VS Code editor. Perfect for the annotated diff viewer. react-diff-view for side-by-side diff component with inline Guardian comments rendered alongside.
Charts
Recharts
Health score trend lines, debt category bars, deploy history. React-native, lightweight, composable. Works perfectly with TanStack Query's real-time data.
Testing
Vitest + React Testing Library + Playwright
Vitest for unit tests. RTL for component tests. Playwright for E2E — critical for testing the staging approval flow and Guardian comment rendering.
Deploy
Vercel
Zero-config React deployment. Preview URLs per PR. You use Codeward to guard your own frontend repo on Vercel — dogfooding from day one.

Backend

Runtime
Node.js 22 + TypeScript
Same language as frontend — one mental model. Node 22 has native TypeScript support. The heavy work happens in sandboxes, not Node itself.
API framework

3× faster than Express. Runs on Cloudflare Workers, Node, Deno — write once. Type-safe routing. The right choice for 2026.

Drizzle is type-safe, lightweight, generates raw SQL — no magic. You see every query. Critical for a product that talks about database quality.

No LangChain. No abstractions. The Anthropic SDK with native tool_use is simpler, faster, cheaper, and you understand every call made.

Build order — what to build, when, and why in that sequence

Build the core loop before the Guardian Agent — here is why

The Guardian Agent has nothing to post without the pipeline running first. It reads real test results, real CVE findings, real duplicate removals. Build the engine first. The Guardian is Layer 2, not Layer 1.

Week 1–2: Project foundations

Monorepo structure · TypeScript strict mode · all env vars set · Drizzle schema pushed · Hono.js server live on Railway · health endpoint returning 200.

Gate: do not proceed until

GET /health returns 200 on Railway URL (not localhost) · all 16 env vars set · 3 DB tables exist in Supabase · Upstash Redis ping passes · R2 test upload works · no secrets committed to git.

Week 2–3: GitHub App + webhook receiver

Register GitHub App · webhook receiver with HMAC verification · BullMQ job created per push · GitHub check run posted (yellow dot on PR).

Gate: do not proceed until

Real push → Railway logs show payload · HMAC rejects forged requests · BullMQ job appears in Upstash · check run shows yellow dot on GitHub PR · returns 200 in under 500ms.

Week 3–4: Ephemeral sandbox engine

Fly.io Machine API · Docker base images with tools (truffleHog, Trivy, k6, tree-sitter) built and pushed · stack detector · test runner inside container · hard 10 min timeout.

Gate: do not proceed until

Fly.io Machine creates, starts, destroys via API · real Node.js repo tests run inside sandbox · machine ALWAYS destroyed (check Fly dashboard for zombies) · timeout fires on infinite loop · total run under 5 minutes.

Week 4–5: First agent — Security Agent + Claude

Security Agent with tool_use · 3 checks first (secrets, CVEs, missing auth) · connects to sandbox · results stored in Postgres · GitHub check run updated green or red.

Gate: do not proceed until

Push with hardcoded API key → blocked → red check run · clean push → green check run with score · findings in Postgres · sandbox destroyed · Claude API cost per run under $0.15 · show to 2 real developers.

Week 5–6: Guardian Agent (FIRST version)

This is the new addition from v2. Build the Guardian now, after the Security Agent works, so it has real data to post.

Guardian reads Security Agent results from Postgres → calls create_pull_request_review via GitHub MCP → posts inline comment on the exact line where the key was found → creates a GitHub Issue for any unresolved finding → submits formal PR review (approve or request_changes).

Gate: do not proceed until

Push with hardcoded key → Guardian posts inline comment on that exact line in the PR diff · Guardian creates GitHub Issue automatically · Guardian submits formal "Request changes" review · PR shows Guardian as a required reviewer · developer can reply to Guardian comment and Guardian replies back.

Week 6–7: AST refactor engine — Bloat Agent

tree-sitter integration (JS/TS first) · semantic duplicate detection · Bloat Agent (Haiku + AST tools) · annotated diff generation · stored in R2 · diff committed back to branch in auto mode · Guardian comment updated with refactor summary.

Week 7–8: React dashboard

Vite + React 18 + TypeScript + Tailwind + shadcn/ui · GitHub OAuth login · Dashboard (runs list + health score) · Live feed (WebSocket stream) · Diff viewer (Monaco + react-diff-view) · Security panel · Guardian activity cards · Deploy to Vercel · Codeward guards its own frontend repo.

Week 9–10: Full pipeline + staging + prod

Staging deploy to Vercel/Render · approval endpoint + UI · GitHub merge trigger · post-deploy monitoring (Sentry) · auto-rollback on error spike · Slack notifications · Guardian creates staging approval card in dashboard.

Week 11–12: User type onboarding + audit mode

Auto-detection of existing vs new repos · "Guard from now" vs "Audit everything first" onboarding screen · full repo clone and scan for audit mode · Guardian creates codeward/audit-fixes branch PR · bulk issue creation for audit findings.

Month 3–4: All agents + Chat Agent

Month 3

Architecture Agent (k6 + EXPLAIN ANALYZE) · Broken Code Agent (flaky tests, type gaps) · AI-Era Agent · Orchestrator (parallel sub-agents) · full debt scoring.

Month 4

Chat Agent (Sonnet + all tools) · Chat UI in React sidebar · Compliance Agent (daily schedule) · Data/DX Agent (weekly) · Health certificate + shareable badge.

Month 5–6

Python + Go AST support · GitLab integration · Self-hosted runner for enterprise · Billing with Stripe · Open source mode with two-tier trust · YC application.

The one rule that decides everything

Ship the webhook → sandbox → Security Agent → Guardian PR comment loop by end of week 5. Get that in front of 5 real engineering teams. The Guardian posting real findings as inline PR comments is the moment Codeward becomes real to a developer — not a dashboard, not a report, but a comment sitting right next to their code saying "this key was found and moved." That moment is what you need to see before building anything else.
