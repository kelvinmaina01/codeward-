Are we building a multi-agent system? Yes. Here is exactly what that means.

The honest answer as an Anthropic engineer

Yes — Codeward is a multi-agent orchestration system. Not one AI that does everything. A coordinator agent that receives the job and dispatches to specialised sub-agents, each with a focused role, specific tools, and the right model for that task. This is how you get both speed and accuracy. A single LLM trying to do security scanning AND AST refactoring AND compliance checking in one prompt would be slow, expensive, and bad at all of them. Specialised agents with tool access are faster, cheaper, and measurably better. This is exactly how Anthropic's own multi-agent research works.

How agents communicate — the message bus

Agents do not call each other directly. They communicate through a shared job queue (BullMQ on Redis) and a structured result store (Postgres). This means:

Orchestrator publishes job
Writes a job record with commitSHA, repoId, priority, and agentList to Redis queue. Each sub-agent picks its own job.
Sub-agents run in parallel
Security, Bloat, Broken Code, Architecture all run simultaneously inside the same sandbox. Average wall-clock time: 4–6 minutes total.
Results aggregated
Each agent writes a structured JSON result to Postgres. Orchestrator reads all results, computes weighted debt score, makes gate decision.

Which model for which agent — the honest reasoning

Orchestrator
Claude Sonnet 4.5 or Claude Opus 4.5
Needs the best reasoning to make gate decisions across complex, conflicting signals. "Should this merge be blocked if security passes but there are 3 architecture issues?" This requires judgment, not just retrieval. Sonnet for cost, Opus for enterprise tier.
Security Agent
Claude Haiku 4.5 + deterministic tools
Most security checks are deterministic tool calls (OWASP ZAP, truffleHog, CVE lookups). The LLM is used to interpret tool output and generate fix suggestions. Haiku is fast and cheap for this pattern.
Bloat Agent
Claude Haiku 4.5 + tree-sitter AST
AST parsing is done by tree-sitter (not the LLM). The LLM reads the structured AST diff and writes the refactored code. Haiku handles this well. Fast + cheap.
AI-Era Agent
Claude Sonnet 4.5
Needs to understand LLM-specific patterns, craft adversarial prompts, reason about RAG pipelines and vector freshness. Requires more capability than Haiku. Sonnet is the right balance.
Compliance Agent
Claude Sonnet 4.5
Legal and compliance reasoning requires nuance. EU AI Act, GDPR, consent versioning — these have edge cases. Sonnet handles the ambiguity better than Haiku.
Chat Agent
Claude Sonnet 4.5
The user-facing agent. Needs to be conversational, technically deep, and able to spawn other agents mid-conversation. Sonnet is the best fit for this pattern — Opus would be too expensive for every chat message.
Important: how you call Claude for agents

You use the Anthropic API with tool_use (function calling). Each agent gets a system prompt defining its role and a list of tools it can call. The agent decides which tools to call, in what order, and how to interpret results. You do NOT prompt the model to write code that calls tools — you define tools as JSON schemas and the model calls them natively. This is the correct pattern. Using claude-sonnet-4-6 for the API calls.

Frontend — React

Framework
React 18 + TypeScript + Vite
React 18 gives you concurrent rendering for real-time live feed updates without UI blocking. Vite over CRA — 10× faster dev builds. TypeScript is non-negotiable for a product that talks about code quality.
Routing
React Router v6 or TanStack Router
TanStack Router if you want type-safe routes (you should). React Router v6 is fine for MVP speed.
Styling
Tailwind CSS v4 + shadcn/ui
shadcn/ui gives you production-quality components (dialogs, tables, dropdowns) without a design team. You own the components — no version lock. Tailwind v4 is significantly faster.
State management
Zustand + TanStack Query
Zustand for UI state (current screen, selected repo, chat messages). TanStack Query for server state (run results, repos, reports) — handles caching, background refetch, and optimistic updates automatically.
Real-time
Socket.io client or Ably SDK
Live run feed needs WebSocket. Socket.io is simplest to self-host. Ably free tier if you want managed.
Code display
Monaco Editor (read-only) + react-diff-view
Monaco is the VS Code editor — perfect for displaying annotated diffs. react-diff-view for the side-by-side diff component.
Charts
Recharts
Health score trend lines, debt category bars, deploy history. Recharts is React-native, lightweight, and composable.
Auth (client)
Better Auth client SDK
Handles GitHub OAuth flow, JWT storage, session refresh. One library, handles everything.
Testing
Vitest + React Testing Library + Playwright
Vitest for unit tests (same config as Vite). RTL for component tests. Playwright for E2E — critical for testing the staging approval flow.
Deployment
Vercel (free hobby tier → Pro)
Zero-config React deployment. Preview URLs per PR — which means you use Codeward to guard your own frontend repo. Dogfood your own product immediately.

External APIs — what they do and exactly how you use them

Anthropic API $5 free credit → pay-per-token
The core of everything. Powers all 8 agents via the /v1/messages endpoint with tool_use. You send a system prompt (the agent's role), the user message (the commit diff or user question), and a list of tools (JSON schemas). The model decides which tools to call. You execute the tools and return results. Repeat until the agent has a final answer.
Free: $5 credit · Sonnet 4.5: $3/$15 per 1M tokens in/out · Haiku 4.5: $0.80/$4 per 1M tokens · Average run cost: ~$0.04–0.12 per commit analysis
GitHub Apps API FREE
The integration layer. Used for: (1) Installing webhooks when user adds a repo. (2) Receiving push/PR events via webhook. (3) Posting check runs (the yellow/green/red dots on PRs). (4) Posting PR comments with the debt report. (5) Triggering merges when staging is approved. You authenticate as a GitHub App using JWT signed with your private key, then exchange for an installation token.
Free · Rate limit: 5000 req/hr per installation · Webhook: unlimited
OWASP ZAP (self-hosted in sandbox) FREE open source
Dynamic security scanning. Runs inside the Firecracker sandbox as a Docker container. You start ZAP in daemon mode, point it at the running app's localhost URL, and it spider-crawls all routes performing OWASP Top 10 checks. Returns structured JSON of findings. Used by the Security Agent to run XSS, CSRF, injection, and auth checks.
Free · Open source · Self-hosted in sandbox · No rate limits
truffleHog v3 (self-hosted) FREE open source
Secrets detection. Scans every file in the repo AND the full git history for 700+ types of secrets (API keys, tokens, passwords, certificates). Runs as a CLI inside the sandbox. Results parsed into structured JSON by the Security Agent. This catches keys committed months ago that no one remembers.
Free · Open source · CLI tool · No rate limits
Trivy (self-hosted) FREE open source
CVE scanning for dependencies AND containers. Scans npm/pip/cargo/gem package lockfiles against the NVD and GitHub Advisory databases. Returns CVE IDs, severity, affected version, and fixed version. Also scans Docker images. Run inside the sandbox by the Security Agent.
Free · Open source · Offline CVE database cached in sandbox image
Grafana k6 (self-hosted) FREE open source
Load testing and performance profiling inside the sandbox. The Architecture Agent writes a k6 script that hits all the customer's API endpoints at 1× and 2× expected load. Measures p99 latency, error rate, and whether the system returns 503 gracefully or crashes. Cold start timing also measured here.
Free · Open source · Runs inside sandbox · No external calls
Supabase FREE 500MB DB + pgvector
Primary database AND vector store. Postgres for all structured data (accounts, runs, reports). pgvector extension for semantic codebase embeddings (used for cross-repo duplicate detection). Built-in auth if you need it. Free 500MB is enough for thousands of runs.
Free: 500MB DB · 2GB bandwidth · 50MB file storage · 500k vector dimensions
Upstash Redis FREE 10k commands/day
Job queue backbone. BullMQ uses Redis as its backing store. Every incoming webhook creates a job in Redis. Every sub-agent job is tracked in Redis. Rate limiting counters stored here. Session tokens cached here. Upstash is serverless Redis — no server to manage.
Free: 10k commands/day · 256MB max · $0.20/100k above free tier
Cloudflare R2 FREE 10GB/month
Object storage for all run artifacts. Every sandbox run produces: a full stdout log, an annotated diff JSON, a security report, and a debt score report. All stored in R2 as JSON files. S3-compatible API — use the AWS SDK with R2 credentials. Zero egress fees (critical — this is why R2 over S3).
Free: 10GB storage · 1M Class A ops · 10M Class B ops · Zero egress
Fly.io Machines API FREE allowance always on
On-demand sandbox VMs. You call the Fly.io Machines API to create a new machine (VM) for each sandbox run, passing your Docker image and environment. Machine starts in under 1 second. You SSH in (or exec) to run your agent tools. When done, you destroy the machine via API. The free allowance covers 3 always-on shared VMs — enough for MVP.
Free: 3 shared-cpu VMs · 160GB bandwidth · $0.0000019/sec for additional machines

Implementation order — exactly what to build, in what order, and why

Think like an Anthropic engineer: build the smallest thing that proves the core value first

The core value of Codeward is: a push goes in, a cleaned, tested, scored commit comes out. Everything else — the dashboard, the chat agent, compliance, DX debt — is built on top of that core loop. Do not build the UI before the loop works. Do not build all 8 agents before one agent works. Build the core loop, show it to 5 real teams, then expand.

Week 1–2 — The webhook loop (proof of concept)

Goal: Push a commit to a test repo → Codeward receives it → runs one agent → posts a result back to GitHub. That's it. Nothing else.

Build
1. GitHub App registration
2. Webhook receiver (Hono.js)
3. HMAC signature verification
4. One BullMQ job created per webhook
5. Security Agent with 3 checks only (secrets, CVEs, missing auth)
6. GitHub check run posted back (pass/fail)
Stack at this point
Node + Hono.js on Railway
BullMQ + Upstash Redis
Supabase Postgres (1 table: runs)
Anthropic API (claude-haiku-4-5)
truffleHog CLI (in Docker)
Trivy CLI (in Docker)

Week 3–4 — The sandbox (core engine)

Goal: Run the full test suite inside a real isolated environment. This is the hardest week. Everything else is built on this foundation.

Build
1. Fly.io Machines API integration
2. Stack detection (package.json, requirements.txt)
3. Docker base images (Node 20, Python 3.11)
4. DB seed with anonymised data
5. Test runner (Jest/pytest inside container)
6. Results parsed and stored
7. Hard timeout at 10 minutes
Do NOT build yet
❌ Dashboard UI
❌ Chat agent
❌ Compliance checks
❌ Slack integration
❌ Auto-refactor
These come later. The sandbox must be rock solid first.

Week 5–6 — The AST refactor engine

Goal: Agent finds duplicate functions, rewrites them, and the tests still pass after.

1. tree-sitter integration (JS/TS first only)
2. Semantic duplicate detection
3. Bloat Agent (Haiku 4.5 + tree-sitter tool)
4. Annotated diff generation (file, line, reason)
5. Re-run test suite after refactor
6. Store diff in R2
7. Post diff as PR comment on GitHub

Week 7–8 — React dashboard (now you have data to show)

Goal: A developer can see their run results in a real UI. Only build what you already have data for.

1. Vite + React 18 + TypeScript + Tailwind + shadcn/ui
2. GitHub OAuth login (Better Auth)
3. Dashboard screen (runs list, health score)
4. Live feed screen (WebSocket stream from BullMQ)
5. Diff viewer screen (read diff from R2)
6. Security panel screen
7. Deploy to Vercel

Week 9–10 — Staging and prod pipeline

Goal: The full push → sandbox → staging → approve → prod loop works end to end.

1. Staging deploy to Vercel/Render (ephemeral per commit)
2. Approval UI in dashboard
3. GitHub merge trigger after approval
4. Post-deploy monitoring (Sentry integration)
5. Auto-rollback on error spike
6. Slack notification for approval requests

Week 11–12 — Codeward AI chat agent

Goal: The chat agent can answer questions about the codebase using real run data.

1. Chat UI in React (sidebar screen)
2. Sonnet 4.5 with tool_use
3. Tools: query_run_history, read_repo_config, explain_debt_item
4. Agent can spawn security scan on demand
5. Conversation history stored in Postgres
6. WebSocket for streaming responses

Month 4+ — Expand agents and debt categories

AI-Era Agent. Data/DX Agent. Health certificate. Shareable safety badge. Billing (Stripe).

The one thing every Anthropic engineer would tell you

Ship the webhook → sandbox → one agent → GitHub check run loop by end of week 2. Then show it to 5 real engineering teams. Get one of them to say "I want this on all my repos." That signal is worth more than 3 months of building in isolation. You will learn more from watching a real team use the basic loop than from building all 8 agents in private.
